上个月，我在公司办公室配置了一台 x86 软路由，以此实现全局代理分流。系统使用的是 iStoreOS，默认情况下它会开启 IPv6，使得内网下的所有终端设备均能获取到物理 IPv6 全局单播地址。虽然绝大多数网络应用在 IPv6 下都能顺畅运行，但我很快发现了一个奇怪的现象：当在电脑微信客户端发送图片或文件时，上传速度经常只有可怜的几十 KB/s，严重干扰了工作效率。

查阅了 [北京电影学院网络信息中心的相关技术分析文章](https://nic.bfa.edu.cn/info/1003/1488.htm?utm_source=chatgpt.com) 后，我找到了症结所在：微信客户端在发送文件时会触发 HTTP-DNS 的优选路由选择，当其与实际网络 IPv6 物理出口发生路由错位时，就会导致速度暴跌。

简单来说：
- 微信发起 DNS 请求时，可能优先通过 IPv6 通道去查询优选服务器的 IP 结果；
- 但实际的文件上传链路却由于网络路由策略被引导到了 IPv4 物理网关（或反之）；
- 一旦用户的 IPv6 出口和 IPv4 出口分别归属于不同的运营商（如多线宽带负载不均），微信就会被迫使用在 IPv6 链路上优选出的目标 IP 去走跨网的 IPv4 通道。这导致延迟飙升和严重的网络丢包，文件上传慢如乌龟。

就像导航告诉你“坐地铁最快”，你却拿着地铁的中转方案去坐了地面公交车，只能到处绕远路。

## 解决办法

1. **临时排查**：在 Windows 网络适配器属性中，临时取消勾选“Internet 协议版本 6 (TCP/IPv6)”。在我的测试中，一旦关掉本机 IPv6，微信文件发送速度瞬间恢复正常水平。但这只能算临时验证。
2. **长期方案**：在软路由 iStoreOS 控制面板中，关闭 LAN 侧的 IPv6 分配（或者将其设为禁用），只保留 IPv4 给内网客户端设备。这有效阻断了终端设备在进行微信优选路由时走入错误出口的可能，从而彻底解决了局域网内微信上传慢的问题。
